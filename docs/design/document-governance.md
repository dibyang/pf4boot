# 设计文档治理

## 问题

历史上 `docs/design` 同时承载当前设计、实施计划、验收记录、一次性修复方案和路线图，文档数量持续增长后，后续开发很难判断哪些内容仍然有效。

## 影响范围

- `docs/design`：保留当前有效的设计事实和开发入口。
- `docs/design/decisions`：沉淀长期架构决策。
- `docs/design/archive`：保存历史计划、验收记录和阶段性路线图。
- `docs/design/en`：保持与中文目录同构，但英文只跟随当前有效设计、长期决策和必要归档索引。

## 分层规则

- 当前设计：描述现行架构、模块边界、运行时契约或开发指南，放在 `docs/design` 顶层。
- 长期决策：记录已经定稿、后续仍需遵守的取舍，放在 `docs/design/decisions`。
- 历史归档：已经完成、被后续设计覆盖、仅用于追溯的计划、验收、路线图和一次性修复方案，放在 `docs/design/archive`。
- 英文翻译：当前设计和长期决策应保持中英文同步；历史归档默认保留既有翻译，不要求后续持续维护。

## 新增文档规则

新增非平凡设计时，优先更新已有当前设计文档。只有满足以下情况时才新增文件：

- 涉及新的公共 API、生命周期、类加载、自动配置、Web/JPA 契约或插件打包边界；
- 需要记录长期决策，且未来开发会反复引用；
- 当前设计文档继续扩写会显著降低可读性。

实施计划和验收清单默认不进入 `docs/design` 顶层。若确实需要记录，应放在 `docs/design/archive`，或在对应 issue、PR、提交说明中维护。

## 迁移规则

整理历史文档时不删除内容，优先移动到归档目录并通过 README 保留入口。若后续确认某个历史文档完全被当前设计覆盖，可以在单独变更中删除，并在提交说明中写明替代文档。

## 验证

文档治理类变更不需要运行 Gradle。完成后应检查：

- `docs/design/README.md` 是否只指向存在的文档；
- 中文文档是否保持 UTF-8；
- 英文 README 是否与中文目录结构一致；
- 归档文档是否仍可通过 `archive/README.md` 找到。
