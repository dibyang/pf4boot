# Repository Guidelines

## Project Shape

This is a Gradle multi-module Java 8 project for `pf4boot`, a PF4J-based Spring Boot plugin framework.

- Core framework modules: `pf4boot-api`, `pf4boot-core`, `pf4boot-starter`, `pf4boot-web-support`, `pf4boot-web-starter`, `pf4boot-jpa`, `pf4boot-jpa-starter`, `pf4boot-jpa-domain-starter`, `pf4boot-actuator`.
- Demo and sample plugin modules live under `samples/*`; the current complex JPA sample is `samples/cross-plugin-jpa`.
- The former root-level demo modules and `app-run` packaging module have been removed.
- Project design notes live in `docs/design`; Chinese is canonical and English translations live in `docs/design/en`. Project constraints live in `docs/constraints`.
- Shared versions live in `gradle.properties`; module membership lives in `settings.gradle`.

## Design Before Coding

- Start non-trivial changes with a short design note before editing production code.
- Put design records under `docs/design/` when the change affects public APIs, plugin lifecycle, class loading, Spring auto-configuration, dependency scopes, packaging, or cross-module contracts.
- Keep design documents in Chinese first, and update the matching English translation under `docs/design/en/` when the Chinese document changes.
- Read `docs/constraints/README.md` before designing or implementing changes.
- Keep the design note practical: problem, affected modules, proposed approach, compatibility impact, verification plan, and open questions.
- For very small mechanical fixes, an inline plan in the conversation is enough; still respect the constraints documented in `docs/constraints`.

## Build And Verification

- Use `.\gradlew.bat` on Windows and `./gradlew` on Unix-like shells.
- Prefer narrow Gradle tasks while iterating, for example `.\gradlew.bat :pf4boot-core:compileJava`.
- Use `.\gradlew.bat build` for whole-project verification when the change affects shared behavior or packaging.
- The root build disables tasks whose names contain `test`, so do not assume `build` runs meaningful tests unless that configuration changes.
- Dependencies resolve from `mavenLocal()`, Maven Central, and the Aliyun public mirror. Network or local Maven state can affect builds.

## Coding Guidance

- Keep Java source compatible with `jdkVersion=1.8`.
- Preserve UTF-8 source encoding and avoid unrelated reformatting.
- Follow the existing package split: public APIs and annotations belong in `pf4boot-api`; runtime implementation belongs in `pf4boot-core`; Spring Boot auto-configuration belongs in starter modules; web/JPA integrations stay in their respective modules.
- Be careful with class loading, bean registration/unregistration, and plugin lifecycle changes. These are framework-level contracts and should be verified with at least a targeted compile or build.
- Do not introduce new framework dependencies casually; prefer the dependency versions already centralized in `gradle.properties`.

## Plugin Module Notes

- Plugin modules apply `net.xdob.pf4boot-plugin`.
- Use `compileOnlyApi` for APIs provided by the host application and `bundle` for dependencies that must be packaged with a plugin.
- Plugin-to-plugin dependencies use `plugin project(":samples:cross-plugin-jpa:...")` or the relevant sample/plugin project path.
- Plugin descriptors can be defined through `pf4bootPlugin { ... }` in module `build.gradle` files or metadata generated by the plugin build.

## Release And Publishing

- Publishable modules are all subprojects except sample modules under `samples/*`.
- Publishing and signing use `signing.properties`; do not edit secrets or require them for ordinary development.
- Release configuration expects the `master` branch and uses tags like `v${version}`.

## Agent Workflow

- Design first, then code. Make the intended module boundary and verification path explicit before changing Java or Gradle files.
- Inspect the relevant module build file before changing dependencies or plugin packaging behavior.
- Check `docs/design` for related prior decisions and update or add a design note when the implementation changes those decisions.
- When adding or changing design documents, update both the Chinese document and the English translation.
- Check `docs/constraints` for project constraints before making API, lifecycle, class loader, packaging, or dependency changes.
- Keep edits scoped to the requested module or behavior; avoid churn in generated Gradle metadata, IDE files, and build outputs.
- If a build cannot be run because dependencies or credentials are unavailable, report exactly which command was attempted and why it could not complete.
